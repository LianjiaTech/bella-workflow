    执行 workflow，没有已发布的 workflow，不可执行。

    ### Request Body
      - `tenantId` (string) Required
        租户ID，取值看右边请求示例
      - `workflowId` (string) Required
        工作流ID，取值看右边请求示例
      - `query` (string) Optional
        Chatflow模式必传，用户的输入/提问
      - `inputs` (object) Required
        允许传入 workflow start 节点定义的各变量值。
        inputs 参数包含了多组键值对（Key/Value pairs），每组的键对应一个特定变量，每组的值则是该变量的具体值。
      - `responseMode` (string) Required
        返回响应模式，支持：
        - `streaming` 流式模式（推荐）。基于 SSE（**[Server-Sent Events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events)**）实现类似打字机输出方式的流式返回。<i>(Chatflow请优先考虑`streaming`模式)</i>
        - `blocking` 阻塞模式，等待执行完毕后返回结果。（请求若流程较长可能会被中断）。 <i>(Chatflow若使用`blocking`模式，最终`outputs`为结束节点绑定变量)</i>
        - `callback` 回调模式，执行任务提交后立即返回，工作流运行完成时回调到指定的`callbackUrl`
        <i>(长时间运行的任务请优先考虑callback模式)</i>
      - `userId` (Long) Required
        用户标识，Ke内必须使用ucid。
      - `userName` (string) Required
        用户名，Ke内用户必须使用与userId相匹配的真实姓名。
      - `callbackUrl` (string) Optional
        回调地址，当 `responseMode` 为 `callback` 时，必填。当workflow执行结束，会以post请求的方式回调callbackUrl。
      - `metadata` (object) Optional
        随路数据，在工作流中可以通过sys.metadata引用，长度不超过4096
      - `flashMode` (int) Optional
        极速模式，设置大于0的值时开启，开启后将不对执行过程持久化，响应时间会更快。默认值为2。  
        - A. 当`flashMode`为1时，工作流执行将不再保存node执行的过程数据。
        - B. 当`flashMode`为2时，A. + 工作流将不再保存输入的query、inputs、metadata入参。
        - C. 当`flashMode`为3时，B. + 工作流将不再保存输出的outputs数据。
        - D. 当`flashMode`为10时，工作流将不再保存所有数据。   
        <i>限制：与挂起不兼容，当工作流存在挂起逻辑时无效</i>
      - `threadId` (string) Optional
        （仅在Chatflow模式下可用）上下文标识（可参考OpenAPI Assistant API中的Thread模块），通过传入上下文标识`threadId`，自定义start节点的sys.thread_id变量
      - `stateful` (boolean) Optional
        （仅在Chatflow模式下可用）状态标识，支持有状态地运行工作流，默认为`false`
        - A. 当`stateful`为`true`时，会将运行工作流时产生的所有`messages`（即所有`直接回复`的内容），自动持久化到`AssistantAPI`对应的`Thread`模块
        - B. 当`stateful`为`false`时，则默认不对`messages`进行处理

    ### Response
    当 `responseMode` 为 `blocking` 时，返回 CompletionResponse object。
    当 `responseMode` 为 `streaming`时，返回 ChunkCompletionResponse object 流式序列。

    ### CompletionResponse
    返回完整的 App 结果，`Content-Type` 为 `application/json` 。
    - `code` (number) workflow状态码
    - `data` (object) 详细内容
      - `workflowRunId` (string) workflow 执行 ID
      - `workflow_id` (string) 关联 Workflow ID
      - `status` (string) 执行状态, `running` / `succeeded` / `failed` / `stopped`
      - `outputs` (json) Optional 输出内容
      - `error` (string) Optional 错误原因

    ### ChunkCompletionResponse
    返回 App 输出的流式块，`Content-Type` 为 `text/event-stream`。
    每个流式块块之间以 `\n\n` 即两个换行符分隔，如下所示：
    <CodeGroup>
    ```streaming {{ title: 'Response' }}
    event:onWorkflowRunSucceeded
    id:onWorkflowRunSucceeded
    data:{"tenantId":"${props.appDetail.tenantId}","workflowId":"${props.appDetail.id}","workflowRunId":"RUN-2407191108200101000002","status":"succeeded","outputs":{"text":"Hello! How can I help?"}}\n\n
    ```
    </CodeGroup>
    流式块中根据 `event` 不同，结构也不同，具体参考右侧的示例。

    ### Errors
    - 400，请求参数错误
    - 500，服务内部异常

<Row>
  <Col>
    ### 功能描述
    此 API 用于调试工作流的触发器，并返回具体的执行结果。

    ### 请求体
    - `workflowId` (string) **必需**
    工作流的 ID。
    - `inputs` (string) **必需**
    工作流运行的输入，格式为 JSON 字符串。
    - `responseMode` (string) **必需**
    API 调用的响应模式。
    - `runVersion` (string) **必需**
    要执行的工作流版本。可以是 **"published"**（已发布）或 **"draft"**（草稿）。
    - `triggerFrom` (string) **固定**
    触发方式。固定为"MANUAL"即手动模式。

    ### 响应
    此 API 调用返回工作流的具体执行结果。
    - `200`，请求成功
    - `400`，请求参数有误

    #### 响应体
    - `tenantId` (string)
    租户 ID。
    - `workflowId` (string)
    工作流 ID。
    - `workflowRunId` (string)
    工作流运行 ID。
    - `status` (string)
    工作流运行状态（例如：“succeeded”、“failed”）。
    - `outputs` (any)
    工作流运行的输出数据。
    - `error` (string)
    如果运行失败，则为错误信息。
  </Col>
